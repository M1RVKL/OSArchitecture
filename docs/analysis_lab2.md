# Аналіз переходу на Шарову Архітектуру та використання DDD

Цей документ містить порівняльний аналіз архітектурних рішень, застосованих у Лабораторній роботі 2, у порівнянні з монолітною структурою Лабораторної роботи 1.

## 1. Що змінилося в структурі проєкту порівняно з лабораторною 1?

У першій лабораторній роботі використовувався підхід «Fat Controllers» (товсті контролери) та «Active Record» (робота безпосередньо з моделями ORM). Вся логіка — валідація запитів, бізнес-правила, розрахунки та звернення до бази даних — була змішана в одному шарі (`controllers`).

У цій роботі проєкт було рефакторено згідно з принципами **Clean Architecture (Шарова архітектура)**. Структуру розділено на 4 чітко ізольовані шари:
*   **Domain:** Містить чисті бізнес-сутності (`Order`, `User`), Value Objects (`Price`, `Address`) та інтерфейси репозиторіїв. Цей шар не має жодних зовнішніх залежностей.
*   **Application:** Містить Use Cases (`AuthService`, `CreateOrderUseCase`), які виконують роль оркестраторів.
*   **Infrastructure:** Реалізує роботу з БД (Prisma). Тут знаходяться імплементації репозиторіїв та Мапери.
*   **Presentation:** Містить тонкі Express-контролери та маршрути. Контролери лише приймають HTTP-запити та передають їх в Application-шар.

## 2. Переваги розділення на шари

1.  **Ізоляція бізнес-логіки:** Правила предметної області (наприклад, статуси замовлень або розрахунок суми) більше не змішані з SQL-запитами чи HTTP-статусами. Це робить код більш передбачуваним.
2.  **Тестованість (Testability):** Тепер ми можемо писати Unit-тести для доменних моделей та Use Cases взагалі без підняття PostgreSQL та Express. Достатньо передати "фейковий" (mock) репозиторій у конструктор.
3.  **Запобігання невалідному стану:** Завдяки Фабрикам та інваріантам всередині доменних сутностей, система фізично не дозволяє створити об'єкт із некоректними даними.
4.  **Дотримання DIP (Dependency Inversion Principle):** Доменний шар диктує правила гри (через інтерфейси-порти), а інфраструктура під них підлаштовується (через адаптери).

## 3. Недоліки та ускладнення

Попри очевидні переваги, впровадження шарової архітектури принесло ряд ускладнень:
1.  **Збільшення об'єму коду (Boilerplate):** Для реалізації найпростішого CRUD тепер потрібно створити мінімум 5 компонентів: Контролер, Use Case, Інтерфейс репозиторію, Реалізацію репозиторію та Мапер.
2.  **Складність мапінгу (Data Conversion Tax):** Дані постійно трансформуються. З БД ми отримуємо `Prisma Object`, Мапер перетворює його на `Domain Entity`, Use Case виконує роботу, після чого Мапер знову перетворює `Domain Entity` на `Prisma Object` для збереження. Це вимагає уважності і ресурсів.
3.  **Ускладнена навігація:** Щоб прослідкувати життєвий цикл одного запиту (traceability), тепер доводиться перемикатися між 4 різними папками, що підвищує когнітивне навантаження на нових розробників.

## 4. Наскільки простіше тепер змінити БД або фреймворк?

Завдяки відокремленню інфраструктури, заміна ключових технологій стала тривіальною задачею:
*   **Заміна Бази Даних (наприклад, з PostgreSQL/Prisma на MongoDB/Mongoose):** Нам **не доведеться** змінювати жодного рядка коду в шарах Domain, Application або Presentation. Усе, що потрібно зробити — написати нову імплементацію для портів (наприклад, `MongoOrderRepository`) та оновити Мапери. Бізнес-логіка цього навіть не помітить.
*   **Заміна Фреймворку (наприклад, з Express.js на Fastify або NestJS):** Зміни торкнуться виключно папки `presentation`. Наші Use Cases та доменні моделі залишаться повністю недоторканими, оскільки вони не залежать від об'єктів `req` та `res`.

## 5. Обґрунтування вибору: Rich Domain Model

Під час проєктування доменного шару наша команда обрала підхід **Rich Domain Model (Багата доменна модель)**, відмовившись від Anemic Domain Model (Анемічної моделі).

**Чому було прийнято це рішення?**
В анемічній моделі сутності є просто "мішками з даними" (getter/setter), а вся логіка лежить у сервісах. Це часто призводить до дублювання перевірок та ризику порушення цілісності даних.

Використовуючи Rich Domain Model (наприклад, у класі `Order`), ми інкапсулювали поведінку всередині самої сутності. Клас `Order` самостійно керує своїм станом через методи `addItem()`, `acceptByRestaurant()`, `assignCourier()`. Він містить жорсткі інваріанти: наприклад, метод `startDelivery()` викине помилку, якщо статус замовлення не `READY` або не призначено кур'єра.

Це гарантує консистентність даних: ніхто (навіть розробник, який пише новий Use Case) не зможе випадково перевести замовлення в нелегальний стан, оскільки сама доменна модель заборонить цю дію.
