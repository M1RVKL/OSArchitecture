# Архітектурний аналіз: Впровадження CQRS

Цей документ містить порівняльний аналіз архітектури системи після рефакторингу та переходу від класичних "товстих" сервісів (God Objects) до патерну розділення відповідальності між командами та запитами (CQRS).

---

## 1. Що змінилося у структурі проєкту?

Головна зміна — це ліквідація монолітної папки `application/use-cases` і розділення логіки на два абсолютно ізольовані потоки:

*   **Write Side (Потік запису):**
    *   `commands/` — містить DTO (об'єкти-наміри) без жодної бізнес-логіки.
    *   `command-handlers/` — містить логіку мутації стану. Звертається до Доменного шару (Entities, Factories) для захисту інваріантів перед збереженням у БД.
*   **Read Side (Потік читання):**
    *   `queries/` — містить параметри для пошуку та фільтрації.
    *   `query-handlers/` — логіка отримання даних. **Повністю ігнорує Доменний шар**, звертаючись напряму до інфраструктури (Prisma) для максимальної швидкодії.
*   **Presentation Layer (Контролери):**
    *   Стали максимально "тонкими". Їхня єдина відповідальність — мапінг HTTP-запиту в об'єкт Command або Query та його передача у відповідний Handler.

---

## 2. Command / Query Handler vs Класичний Service

У попередній лабораторній один клас (наприклад, `RestaurantService`) створював ресторани, шукав їх і деактивовував. Це грубо порушувало **Single Responsibility Principle (SRP)**.

| Характеристика | Класичний Service | Command / Query Handlers |
| :--- | :--- | :--- |
| **Відповідальність** | Змішана (читання + запис) | Суворо розділена (одна дія = один клас) |
| **Розмір файлу** | Схильний до розростання ("God Object") | Компактний, має лише один метод `execute()` |
| **Зв'язність (Coupling)** | Висока (зміни у вибірці можуть зламати логіку збереження) | Мінімальна (повна ізоляція операцій) |
| **Тестування** | Складне (потребує мокування великої кількості залежностей) | Просте та ізольоване |

---

## 3. Переваги CQS

Впровадження розділення принесло команді чотири ключові переваги:

1.  **Оптимізація швидкодії читання:** Query Handlers йдуть в обхід важкого доменного шару. Ми більше не створюємо складні сутності (`Order`, `Address`, `Price`) лише для того, щоб віддати їх у форматі JSON. Дані забираються з бази і віддаються клієнту напряму.
2.  **Явність намірів системи:** Замість абстрактного `updateOrderStatus(...)` система тепер оперує бізнес-термінами: `AssignCourierCommand`, `CancelOrderCommand`. Код став самодокументованим.
3.  **Ідеальне тестове середовище:** Операції запису (Commands) тепер тестуються миттєвими Unit-тестами (через In-Memory репозиторії), а операції читання (Queries) — надійними Integration-тестами з реальною БД.
4.  **Паралельна розробка:** Архітектура дозволила розпаралелити роботу без конфліктів у системі контролю версій: один розробник фокусувався на мутаціях і доменних правилах, інший — на вибірці даних.

---

## 4. Недоліки CQS

Жодна архітектура не є ідеальною. Ми зіткнулися з такими компромісами:

*   **Boilerplate (Збільшення об'єму коду):** Замість додавання одного методу в існуючий сервіс, тепер необхідно створювати мінімум 2-3 нові файли (DTO команди, сам Handler, іноді кастомний Response DTO).
*   **Складність навігації:** Для новачків потік виконання (`Route -> Controller -> Command -> Handler -> Domain -> Repository`) виглядає більш роздробленим і складним для відстеження, ніж виклик одного "всемогутнього" сервісу.

---

## 5. Вплив на розширюваність

CQRS ідеально підтримує принцип відкритості/закритості (SOLID: OCP).

*   **Як було раніше:** Щоб додати нову фічу, доводилося **модифікувати** існуючий клас сервісу, ризикуючи зламати старий код.
*   **Як стало:** Щоб додати нову фічу, ми просто **додаємо** два нові файли (наприклад, `MarkOrderAsReadyCommand` та її Handler). Старий код при цьому взагалі не змінюється, а отже, не потребує повторного тестування.

---

## 6. Query Data Structure vs Доменна модель

Найважливіше архітектурне рішення — це відмова від використання доменних моделей для повернення результатів клієнту.

*   **Доменна модель (напр., `Order`)** — це **Rich Model** (Багата модель). Вона інкапсулює поведінку, перевіряє інваріанти, використовує Value Objects і містить поля, які не можна показувати назовні (наприклад, `passwordHash` у `User`).
*   **Query Model (Read Model)** — це **Anemic Data Structure** (Анемічна структура). Це простий, плоский JSON-об'єкт. У ньому немає жодної логіки чи методів. Він ідеально підігнаний під конкретний екран UI клієнта. 

**Висновок:** Клієнту не потрібні наші складні доменні правила для простого відображення списку. Розділення цих моделей гарантує безпеку даних та високу продуктивність.
